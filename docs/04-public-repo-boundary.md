# Public Repo Boundary

## What should be public

Include:
- the standalone `product-design-from-lenny` skill
- safe reusable references
- installation and adaptation docs
- public-safe examples
- design rationale and build-process notes

## What should stay private

Exclude:
- personal workspace files
- assistant memory files
- unrelated private projects
- customer or business-sensitive material
- raw data that creates storage or licensing problems

## Rule of thumb

If a file teaches the public skill, keep it.
If a file leaks context from the private workspace, remove it.

## Publishing decision rule

Before publishing, ask:

1. Can this file be safely shared?
2. Does it help someone use or understand the skill?
3. Is it specific to this public project rather than the private monorepo?

If the answer is not clearly yes, do not publish it.
