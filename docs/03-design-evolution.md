# Design Evolution

## Version story

This public project comes from a broader wave of experiments around Lenny-derived product skills.
Over time, the build process converged around a few durable lessons.

## Lesson 1: boundaries matter more than cleverness

It is easy to make a skill feel ambitious by expanding scope.
It is much harder to make a skill reusable.

The durable pattern was to narrow the job:
- focus on product planning
- focus on feature design
- avoid turning the skill into a giant everything-tool

## Lesson 2: distilled method beats raw source overload

If you pour too much raw source material into a skill, the result becomes heavy and unstable.
A better pattern is:

1. source material
2. distilled notes
3. skill instructions
4. examples

## Lesson 3: process docs are part of the product

For a public project, the process is not fluff.
It helps others:
- trust the structure
- understand the tradeoffs
- remix the approach for their own agent systems

## Lesson 4: portability is a feature

A skill tied to only one runtime is less useful.
This project is intentionally structured so it can be used in:
- OpenClaw
- Claude Code
- Codex-style IDE workflows
- other file-based agent setups

## Public-repo interpretation

The public version should show both:
- the final skill artifact
- the thinking path that produced it
